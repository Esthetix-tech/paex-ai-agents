## 3. Why Token / Secret Governance Candidate Review Is Needed

Future automation cannot safely rely only on command wording or actor attribution.

Actor attribution does not establish token ownership.

K authorization evidence does not establish credential ownership.

Head SHA lock does not establish credential trust.

Token name, secret name, environment variable name, bot name, GitHub App name, or API key reference does not prove ownership or authority.

Secret reference is not the same as secret value.

Secret name is not the same as secret owner.

Token name is not the same as token owner.

GitHub App installation does not prove K authorization.

Bot credential does not prove K authorization.

Codex output does not prove credential authorization.

Token / secret / API key / webhook secret ownership must not be inferred by AUTO-10.

Credential owner, credential actor, merge actor, execution actor, bot actor, and GitHub App actor may differ.

Credential ambiguity can create false positive and false negative governance risks.

AUTO-10 remains candidate-only until separately reviewed and canonicalized if ever authorized.

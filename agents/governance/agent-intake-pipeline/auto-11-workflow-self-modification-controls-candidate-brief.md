## Why Workflow Self-modification Controls Candidate Review Is Needed

Future automation cannot safely rely only on the existence of a workflow file, workflow trigger, GitHub Actions actor, or job permission.

Workflow files can modify future execution conditions if they add or change triggers, permissions, scripts, command parsers, automation configuration, or runtime configuration.

Workflow self-modification risk is separate from protected path risk, wording risk, K authorization evidence, head SHA alignment, actor attribution, and token / secret governance.

Workflow metadata does not create K authorization.

Workflow file names do not create K authorization.

Workflow triggers do not create K authorization.

GitHub Actions actor identity does not create K authorization.

GITHUB_TOKEN permission does not create K authorization.

Workflow permission or job permission does not create production authority.

Guard scripts, validation scripts, command parsers, and PR Guard rule references can become bypass points if they can modify themselves or the controls that judge them.

Recursive automation, self-approval, and self-merge can collapse executor, reviewer, and approver boundaries.

AUTO-11 remains candidate-only until separately reviewed and canonicalized if ever authorized.
